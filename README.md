## 📌 Projects

### ⚽ BallKnowledge &nbsp;·&nbsp; [Repo](https://github.com/ilkkiinn/BallKnowledge)
> *JavaScript · Social Logging Platform · Frontend*

A Letterboxd-style platform for soccer — built for fans who want to log matches they've watched, track their history, and keep a record of games that meant something to them. The core insight was that soccer culture is deeply personal and opinionated, but there's no good tool for capturing that beyond a notes app. BallKnowledge fills that gap.

**Key decisions:** User-centric logging flow, match search and tagging, clean profile layout that puts your history front and center.

---

### 📊 DealOrNoDealNBA &nbsp;·&nbsp; [Repo](https://github.com/ilkkiinn/DealOrNoDealNBA)
> *HTML · Trade Evaluator · Data-Driven UI*

NBA trade evaluator built on the premise that most fan trade discussions lack any real framework. This gives it one. Users input players on both sides of a trade and get a structured verdict based on value metrics. The interesting part wasn't the UI — it was deciding what "a good trade" even means and encoding that logic consistently.

**Key decisions:** Clear evaluation criteria, clean input/output flow, designed for people who actually follow the league.

---

### 📚 Zema &nbsp;·&nbsp; [Repo](https://github.com/ilkkiinn/Zema)
> *HTML/CSS · Motivational Study Site · Real-World Use Case*

Built for a real person with a real exam coming up. The goal was simple: a focused, distraction-free site that keeps someone motivated when studying gets hard. No over-engineering — just clean layout, good copy, and something that actually gets used. Shipping for a real user (not just yourself) changes how you think about every design decision.

**Key decisions:** Minimal interface, content-first layout, built around the user's actual needs.

---

### ✅ ebk-checker &nbsp;·&nbsp; [Repo](https://github.com/ilkkiinn/ebk-checker)
> *TypeScript · Validation Tool · Typed Architecture*

A TypeScript-based validation tool for basketball knowledge content. The point of using TypeScript here wasn't just syntax — it was about making the data structure explicit and catching errors before they happen. When you're validating user input against a defined schema, type safety isn't optional.

**Key decisions:** Strict typing throughout, single-responsibility design, built to be extended without rewriting core logic.

---

### 🍽 suadarestaraunts &nbsp;·&nbsp; [Repo](https://github.com/ilkkiinn/suadarestaraunts)
> *HTML/CSS · Restaurant Showcase · Layout & Design*

A restaurant directory and showcase site focused on presentation. The work here was mostly about layout — how do you make content-heavy pages feel organized without being rigid? Clean grid structure, consistent typography, and letting the content breathe.

**Key decisions:** No unnecessary JavaScript, semantic HTML structure, mobile-considered layout.

---

### 🔒 focus-guardian &nbsp;·&nbsp; [Repo](https://github.com/ilkkiinn/focus-guardian) &nbsp;·&nbsp; *(Forked & Contributed)*
> *Python · AI Productivity Tool · Hackathon*

Hackathon project built around Gemini AI — blocks distracting sites, enforces timed deep work sessions, and delivers personalized coaching through the model. I owned the frontend and core interaction logic: designing the session UI, wiring up the blocking mechanism to the timer state, and making sure the AI responses surfaced in a way that felt useful rather than just noise. Tight deadline, real constraints, shipped something that worked.

**Key decisions:** Gemini API for contextual coaching, component-level state tied to session lifecycle, UI designed to stay out of the way while you're focused.
